# 多条流的组合

## 到目前为止的局限

之前处理的都是**单条流**的故事：一个输入源 → 一串操作符 → 一个输出。

但真实业务里经常需要多条流协作：

- 保存按钮和 Ctrl+S 都能触发保存 → **多个来源合成一条流**
- 两个下拉框的选择共同决定表格数据 → **多条流的最新值组合在一起**
- 页面初始化要同时加载用户信息、菜单、配置 → **多条流并行，全部完成后一起用**

这些问题用单条流不好解决——你需要把多条流"拼"起来。

RxJS 提供了几个组合操作符，它们的区别就一句话：**拼的方式不同**。

---

## `merge` —— 谁来了就转发谁

最简单的组合方式：把多条流直接混在一起，谁先到就先出。

```js
const { merge, fromEvent } = rxjs;
const { map, filter } = rxjs.operators;

// 三种方式都能触发"关闭弹窗"
const closeBtn$ = fromEvent(closeBtn, "click").pipe(map(() => "close"));
const overlay$ = fromEvent(overlay, "click").pipe(map(() => "close"));
const escKey$ = fromEvent(document, "keydown").pipe(
  filter((e) => e.key === "Escape"),
  map(() => "close"),
);

// 合并成一条流，下游不关心"用户是怎么关的"
merge(closeBtn$, overlay$, escKey$).subscribe(() => {
  dialog.style.display = "none";
});
```

`merge` 不等、不组合、不加工——纯转发。适合**多个入口触发同一个动作**。

上一课 concatMap 的 demo 里用 Subject 实现了"按钮 + 快捷键都能保存"，其实用 `merge` 也能做：

```js
const btnSave$ = fromEvent(btn, "click").pipe(map(() => getConfig()));
const keySave$ = fromEvent(document, "keydown").pipe(
  filter((e) => (e.ctrlKey || e.metaKey) && e.key === "s"),
  tap((e) => e.preventDefault()),
  map(() => getConfig()),
);

merge(btnSave$, keySave$)
  .pipe(concatMap((config) => saveConfig(config)))
  .subscribe();
```

## `combineLatest` —— 谁变了都重新算

`merge` 是"多选一"——多个来源独立输出。`combineLatest` 是"多合一"——把多条流的**最新值打包在一起**，任何一条变化都重新组合。

```js
const { combineLatest, fromEvent } = rxjs;
const { map, switchMap, startWith, catchError } = rxjs.operators;

const country$ = fromEvent(countrySelect, "change").pipe(
  map((e) => e.target.value),
  startWith("all"), // 给初始值，否则 combineLatest 不会触发
);
const category$ = fromEvent(categorySelect, "change").pipe(
  map((e) => e.target.value),
  startWith("all"),
);

combineLatest([country$, category$])
  .pipe(
    switchMap(([country, category]) =>
      fetch(`/api/report?country=${country}&category=${category}`).then((r) =>
        r.json(),
      ),
    ),
  )
  .subscribe((data) => renderTable(data));
```

### combineLatest 的关键行为

1. **所有流都至少发过一次值后，才开始输出。** 如果 `country$` 还没选，即使 `category$` 变了也不会触发——所以上面用 `startWith` 给初始值。
2. **之后任何一条流变化，都会拿各自的最新值重新组合输出。**

### 和 `merge` 的区别

- `merge([a$, b$])` → 输出的是**单个值**，a 来就输出 a 的值，b 来就输出 b 的值
- `combineLatest([a$, b$])` → 输出的是**数组**，永远是两个值打包在一起 `[a最新, b最新]`

## `withLatestFrom` —— 主流触发，顺便带上副流的值

`combineLatest` 是"谁变了都触发"。但有时候你只想在**某一条流发值时才触发**，另一条流只是被动提供当前值。

```js
const { withLatestFrom } = rxjs.operators;

const click$ = fromEvent(submitBtn, "click");
const formValue$ = fromEvent(form, "input").pipe(
  map(() => Object.fromEntries(new FormData(form))),
  startWith(Object.fromEntries(new FormData(form))),
);

click$
  .pipe(
    withLatestFrom(formValue$),
    switchMap(([_, formData]) => submitForm(formData)),
  )
  .subscribe((result) => showResult(result));
```

用户每敲一个字，`formValue$` 都在更新——但不会触发提交。只有**点击按钮**的那一刻，才去读 `formValue$` 的最新值，一起传给下游。

### `combineLatest` vs `withLatestFrom`

- `combineLatest([click$, formValue$])` → 用户每敲一个字**都会触发**（因为 `formValue$` 变了）——不是你想要的
- `click$.pipe(withLatestFrom(formValue$))` → **只有点击才触发**，`formValue$` 只是被动提供值

---

## `forkJoin` —— 全部完成后，一次性打包

`combineLatest` 是持续响应的——流只要变了就输出。而 `forkJoin` 只在所有流**都完成后**，输出一次就结束。跟 `Promise.all` 一个意思。

```js
const { forkJoin, from } = rxjs;

forkJoin({
  user: from(fetch("/api/user").then((r) => r.json())),
  menus: from(fetch("/api/menus").then((r) => r.json())),
  config: from(fetch("/api/config").then((r) => r.json())),
}).subscribe(({ user, menus, config }) => {
  initApp(user, menus, config);
});
```

**forkJoin 的坑：** 如果某个内部流**永远不 complete**（比如 `interval`），`forkJoin` 就永远不会输出。它只适合一次性的流（HTTP 请求、读文件等）。

### 加上错误处理

```js
forkJoin({
  user: fetchUser().pipe(retry(1)),
  menus: fetchMenus().pipe(retry(1)),
  config: fetchConfig().pipe(
    catchError(() => of({ theme: "default" })), // 配置加载失败就用默认的
  ),
})
  .pipe(
    catchError((err) => {
      showErrorPage("初始化失败");
      return EMPTY;
    }),
  )
  .subscribe(({ user, menus, config }) => {
    initApp(user, menus, config);
  });
```

config 失败了给个默认值继续跑，user 和 menus 失败了整体报错。**部分降级优于整体失败。**

---

## 速记

| 操作符           | 处理                             | 输出次数       |
| ---------------- | -------------------------------- | -------------- |
| `merge`          | 多条流混在一起，谁来转发谁       | 持续输出       |
| `combineLatest`  | 谁变了都重新组合所有最新值       | 持续输出       |
| `withLatestFrom` | 主流触发时，顺便带上副流的当前值 | 主流发值时输出 |
| `forkJoin`       | 等全部完成，打包输出一次         | 只输出一次     |

- **多个来源触发同一个动作** → `merge`
- **多个条件联动查询** → `combineLatest`（记得加 `startWith`）
- **主事件 + 附带上下文** → `withLatestFrom`
- **页面初始化并行请求** → `forkJoin`

---

## `startWith`

前面用了 `startWith`，单独说一下。它的作用是：**在流订阅后、还没有真正发值之前，先发一个初始值。**

```js
const search$ = fromEvent(input, "input").pipe(
  map((e) => e.target.value),
  startWith(""), // 一订阅就先发一个空字符串
);
```

它不是 `combineLatest` 专用的，但最常跟 `combineLatest` 一起出现——因为 `combineLatest` 要求每条流都发过值才开始输出，`startWith` 正好解决"还没操作过就没有初始值"的问题。

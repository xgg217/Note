# Subject 与共享

前面讲了怎么把多条流组合起来。这里反过来——**一条流的数据，怎么让多个地方用？**

具体来说有下面的场景：

1. **从外部往流里塞值**：比如按钮点击、命令式代码调用——怎么把这些"事件"接入 RxJS 管道？
2. **一份数据多处订阅**：比如用户信息被导航栏、侧边栏、权限组件同时需要——怎么避免重复请求？

这两个问题看起来不相关，但它们共享一个底层概念：**流的冷和热**。

---

## 冷流和热流

> 参考链接（官方出处）
>
> - RxJS 官方术语表（直接定义了 Cold / Hot）：https://rxjs.dev/guide/glossary-and-semantics
> - ReactiveX 官方文档（独立章节讲 Hot and Cold Observables）：https://reactivex.io/documentation/observable.html#hot-and-cold-observables
>
> 简要说明："冷流 / 热流"不是民间说法，而是 ReactiveX 体系的标准术语。RxJS 官方术语表对 Cold 和 Hot 有明确定义：冷流在每次订阅时才创建数据源，热流的数据源在订阅之前就已存在。

Observable 是懒的，subscribe 一次才执行一次。这就是**冷流**。

```js
// 冷流：每次 subscribe 都独立执行
const user$ = rxjs.defer(() => fetch("/api/user").then((r) => r.json()));

user$.subscribe((u) => console.log("A:", u)); // 发请求 1
user$.subscribe((u) => console.log("B:", u)); // 发请求 2 ← 又发了一次
```

冷流就像自动售货机——每个人按按钮都会各自出一瓶水。

**热流**正好相反：数据源跟 subscribe 无关，一直在推。订阅只是"接上"而已。

```js
// 热流：WebSocket 消息一直在推，不管有没有人订阅
const ws = new WebSocket("wss://example.com/prices");
const price$ = fromEvent(ws, "message");
// 你 subscribe 之前推过来的股价，你永远收不到
```

热流像广播电台——你打开收音机只能听到当前正在播的内容，之前播过的你收不到。

**我们这里讲解的API，本质上都是在做一件事：控制流的"冷热"。**

比如，你可以这样理解：

- **Subject** 天生就是热流——它不是"subscribe 时才产生数据"，而是你随时 `.next()` 往里推，所有当前订阅者共享同一份推送。没人订阅时推的值就丢了，跟广播电台一样。
- **shareReplay** 是把一条冷流"加热"——第一个人订阅时才启动上游，但后续订阅者共享结果而不会重复执行。

换句话说：Subject 是**原生的热流**，shareReplay 是**把冷流变热的工具**。理解了这一点，下面每个 API 就很好记了。

---

## Subject —— 自己能控制的事件源

普通 Observable 的数据从哪来，在创建时就定好了（`fromEvent`、`interval`、`of`...）。你没法从外面往里塞值。

Subject 打破了这个限制：**它既能被订阅，又能从外面 `.next()` 塞值进去。**

从冷热的角度看，Subject 就是一个**热流**——它不因为有人订阅才开始跑，而是你什么时候 `.next()`，什么时候就推值。多个订阅者收到的是同一份数据，不会各跑各的。

```js
const { Subject } = rxjs;

const action$ = new Subject();

// 当 Observable 用：订阅它
action$.subscribe((val) => console.log("收到:", val));

// 当 Observer 用：手动塞值
action$.next("刷新"); // 控制台：收到: 刷新
action$.next("重试"); // 控制台：收到: 重试
```

### 典型场景：通知中心

先看不用 RxJS 怎么写。需求很简单：按钮能刷新通知列表，标记已读后也要刷新。

```js
// 不用 RxJS：手动管理刷新逻辑
async function loadNotifications() {
  try {
    const list = await fetch("/api/notifications").then((r) => r.json());
    renderNotifications(list);
  } catch {
    renderNotifications([]);
  }
}

// 初始加载
loadNotifications();

// 按钮刷新
refreshBtn.addEventListener("click", () => loadNotifications());

// 标记已读后刷新
async function markAsRead(id) {
  await fetch(`/api/notifications/${id}/read`, { method: "POST" });
  loadNotifications();
}
```

这段代码能跑，如果有防抖、竞态、重试等等问题呢？其实也好解决：封装一下不就行了？加个防抖包一层、错误处理包一层，传统写法照样能搞定。没错，**从功能上讲，传统写法什么都能实现**。

但这里想说的不是"谁好谁坏"，而是一个**思维方式的区别**。

传统写法的思路是**"调用"**——你在每个触发点写代码去调一个函数。逻辑的入口、控制流程、错误处理，都是由调用方来负责或协商的。你封装得再好，本质上还是"我调你、你调他"的命令链。当触发源从 2 个变成 5 个、控制逻辑从防抖变成"防抖 + 竞态取消 + 失败重试"，你会发现封装层越裹越厚，调用方和被调用方的职责边界越来越模糊。

Subject 代表的是另一种思路：**"推送"**。触发方不调用任何函数，只做一件事——往一个入口推一个信号。至于这个信号怎么被处理、要不要防抖、要不要取消旧请求，触发方完全不知道、也不需要知道。处理逻辑全部集中在管道里，跟触发方零耦合。

这就是从**"主动调用"到"被动响应"**的思维转换。不是说响应式一定比命令式好——命令式直观、好调试、学习成本低，很多场景用它就够了。但当你的系统里**触发源越来越多、控制需求越来越复杂**的时候，"声明一条管道，让数据自己流过去"比"在每个入口写调用逻辑"更容易维护。这是一种架构层面的解耦能力。

理解了这个区别，再看 Subject 的写法——**把"谁来触发"和"触发后怎么处理"彻底分开**：

```js
const refresh$ = new Subject();

// 把 Subject 接入管道
refresh$
  .pipe(
    startWith(null), // 一订阅就先加载一次
    switchMap(() => fetch("/api/notifications").then((r) => r.json())),
    catchError(() => of([])),
  )
  .subscribe((list) => renderNotifications(list));

// 任何地方都能触发刷新
refreshBtn.addEventListener("click", () => refresh$.next());

async function markAsRead(id) {
  await fetch(`/api/notifications/${id}/read`, { method: "POST" });
  refresh$.next(); // 标记已读后，重新加载列表
}
```

`refresh$` 是一个"入口"——按钮能往里推，业务代码也能往里推。下游管道不关心"谁触发的"，只关心"该刷新了"。所有的控制逻辑（防抖、竞态取消、错误兜底）都集中在管道里，加需求只改一处。

### Subject 的定位

**Subject 是命令式世界和响应式世界之间的桥梁。**

什么意思？**命令式**就是"你来指挥"——第几步做什么、什么时候调什么函数，流程由你控制。按钮回调、`async/await`、生命周期钩子，这些都是命令式代码。**响应式**正好反过来——你只声明"数据来了怎么走"，至于数据什么时候来、来几次，你不管。`pipe(switchMap(...), catchError(...))` 这种管道就是响应式的。

两种风格各管各的没问题，但实际项目里它们一定会碰头：按钮点了要触发请求管道、表单提交后要刷新列表流、WebSocket 断线后要通知重连逻辑。这时候你需要一个东西，**一面朝着命令式代码接收指令，一面朝着响应式管道推送数据**——Subject 就是这个东西。`.next()` 是命令式的入口，`.pipe()` 是响应式的出口，两头一接，桥就搭好了。

所以 Subject 的正确用法是做流的"入口"——把外部事件接入管道。

**注意：**不要在 `pipe()` 内部去 `.next()` 一个 Subject，那大概率是设计有问题——管道内部本身就是响应式的，不需要桥梁。

## BehaviorSubject —— 记住最新值

普通 Subject 有个问题：subscribe 之前推过的值，新订阅者拿不到。

```js
const sub = new Subject();
sub.next("hello"); // 这个值发出去了，但没人在听

sub.subscribe((v) => console.log(v)); // 订阅了，但 "hello" 已经错过了
```

`BehaviorSubject` 解决这个问题——它**在热流的基础上加了一层缓存**，会记住最新值，新订阅者一上来就能拿到：

```js
const { BehaviorSubject } = rxjs;

const currentUser$ = new BehaviorSubject(null); // 必须给初始值

currentUser$.subscribe((u) => console.log("导航栏:", u));
// 立刻输出：导航栏: null

currentUser$.next({ name: "Tom", role: "admin" });
// 输出：导航栏: { name: "Tom", role: "admin" }

// 后来的订阅者也能立刻拿到最新值
currentUser$.subscribe((u) => console.log("侧边栏:", u));
// 立刻输出：侧边栏: { name: "Tom", role: "admin" }
```

**什么时候用 BehaviorSubject：** 你需要表示一个"当前状态"时——当前登录用户、当前选中的 tab、当前主题。这些东西任何时候问都应该有值，不存在"还没初始化"的情况。

从冷热角度看：**它是热流 + 缓存 1 个值**。热流保证多个订阅者共享同一份推送，缓存保证后来者不会漏掉当前状态。

---

## ReplaySubject —— 缓存最近 N 个值

`BehaviorSubject` 固定缓存 1 个，且必须给初始值。`ReplaySubject` 更灵活：

```js
const { ReplaySubject } = rxjs;

const log$ = new ReplaySubject(3); // 缓存最近 3 个

log$.next("日志A");
log$.next("日志B");
log$.next("日志C");
log$.next("日志D"); // A 被挤掉了

log$.subscribe((v) => console.log(v));
// 日志B
// 日志C
// 日志D
```

**什么时候用 ReplaySubject：** 比如，聊天消息回放、操作日志——新打开的面板需要看到最近几条历史记录。

从冷热角度看：**它是热流 + 缓存 N 个值**。和 BehaviorSubject 一样共享推送，只是缓存容量更大、不需要初始值。

### 三种 Subject 速记

| 类型              | 缓存      | 要初始值吗 | 典型场景                         |
| ----------------- | --------- | ---------- | -------------------------------- |
| `Subject`         | 不缓存    | 不要       | 纯事件触发（按钮点击、手动刷新） |
| `BehaviorSubject` | 最新 1 个 | 要         | 当前状态（用户信息、选中项）     |
| `ReplaySubject`   | 最近 N 个 | 不要       | 历史回放（聊天记录、日志）       |

## `shareReplay` —— 让冷流变成共享的

回到开头的问题：一份数据被多处订阅，冷流会导致重复请求。

Subject 是"从零开始就是热的"，但很多时候你的数据源本身是冷流（比如 HTTP 请求）。你不想手动搞一个 Subject 去转发，只想让这个冷流"变热"——`shareReplay` 就是干这个的。

```js
// 冷流：三次 subscribe → 三次请求
const user$ = rxjs.defer(() => fetch("/api/user").then((r) => r.json()));

user$.subscribe((u) => renderNav(u)); // 请求 1
user$.subscribe((u) => renderSidebar(u)); // 请求 2
user$.subscribe((u) => checkRole(u)); // 请求 3
```

加上 `shareReplay` 就解决了：

```js
const { shareReplay } = rxjs.operators;

const user$ = rxjs
  .defer(() => fetch("/api/user").then((r) => r.json()))
  .pipe(shareReplay({ bufferSize: 1, refCount: true }));

user$.subscribe((u) => renderNav(u)); // 请求 1（第一个订阅者触发请求）
user$.subscribe((u) => renderSidebar(u)); // 共享上面的结果
user$.subscribe((u) => checkRole(u)); // 共享上面的结果

// 甚至 5 秒后再订阅，也能直接拿到缓存
setTimeout(() => {
  user$.subscribe((u) => console.log("延迟订阅:", u));
}, 5000);
```

`shareReplay` 做了两件事：

1. **共享**：多个订阅者共享同一个上游订阅，不重复执行
2. **缓存**：记住最近的值，后来的订阅者直接拿到

`refCount: true` 表示所有订阅者都取消后，上游也取消（释放资源）。

### shareReplay 不是性能优化按钮

**注意：shareReplay在省请求的同时，也改变了数据语义**——后来的订阅者拿到的可能是缓存里的旧值，而不是重新请求的新值。

所以加之前要想清楚两件事：

- 这份数据**真的会被多处订阅**吗？只有一个订阅者就不需要 share，白加反而埋坑。
- 缓存的数据**业务上会不会变旧**？`shareReplay` 本身**不会自动让缓存过期**——它只管"记住"，不管"这个值现在还准不准"。比如用户信息 5 分钟前拿的，权限可能已经变了，但缓存里还是旧的，后来的订阅者照样拿到旧值。

一句话总结：**shareReplay 帮你记住旧值，但不帮你判断旧值还对不对。** 数据什么时候该刷新，需要你自己来定——比如定时重新请求、用户操作后手动刷新、登出时清空，等等。

## 流的"温度"总结

把这一章的概念串起来，流的温度其实是一个连续谱：

```
冷                                                热
│                                                  │
│  defer/Observable       shareReplay       Subject/BehaviorSubject
│  每次 subscribe         共享+缓存          独立的事件源
│  都独立执行             后来者也能拿到      跟 subscribe 无关
```

- **冷流**：每次订阅独立执行——适合"每次都要最新结果"
- **shareReplay（冷→热）**：第一次订阅时执行，后续共享缓存——适合"多处用同一份数据"
- **热流**（Subject 系列）：数据源跟订阅无关——适合"外部事件入口"

## Promise 和 Observable 互转

实际项目里经常需要两者互转，简单记住就行。

### Promise → Observable

```js
// from：包装一个已经在执行的 Promise
const user$ = rxjs.from(fetch("/api/user").then((r) => r.json()));

// defer：每次订阅时才执行 Promise（延迟执行）
const user$ = rxjs.defer(() => fetch("/api/user").then((r) => r.json()));
```

在 `switchMap` / `mergeMap` 内部直接返回 Promise 就行，RxJS 会自动转——不用手动包 `from`：

```js
source$
  .pipe(
    switchMap((keyword) =>
      fetch(`/api/search?q=${keyword}`).then((r) => r.json()),
    ),
  )
  .subscribe();
```

### Observable → Promise

```js
const { firstValueFrom } = rxjs;
const user = await firstValueFrom(user$);
```

### 什么时候需要互转

通常在**两种编程方式的边界**处。比如你的数据管道是 RxJS，但框架的生命周期钩子是命令式的——就需要在钩子里 subscribe，在请求部分用 `from` 包装。

**管道内部应该保持纯 Observable**，不要 pipe 到一半突然 `await`。

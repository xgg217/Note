# 基础

## 设计意图

+ 单条流只是起点
+ 真实的 UI 是多个数据源的交汇——用户的点击、服务器的推送、定时器的心跳、路由的变化
+ 组合操作符让你声明式地表达"这几条流之间的关系"，而不是手动维护一堆状态变量和标志位

## 操作符

+ 真实业务中，数据很少只来自一个源头。组合操作符解决的是**多条流之间的协调**问题

  | 操作符                    | 用途                                     | 典型场景             |
  | ------------------------- | ---------------------------------------- | -------------------- |
  | `merge(a$, b$)`           | 合并多条流，谁来了就发谁                 | 来自多个按钮的事件   |
  | `concat(a$, b$)`          | 串接流：前一条完了才订阅后一条           | 按顺序执行多个请求   |
  | `combineLatest([a$, b$])` | 任一流更新时，取所有流的最新值           | 表单联动、多条件筛选 |
  | `forkJoin([a$, b$])`      | 等所有流 complete，取每条流的最后值      | 并行请求等全部完成   |
  | `zip(a$, b$)`             | 按位置一一配对                           | 严格的一对一对应     |
  | `race(a$, b$)`            | 取最先发出值的流，丢弃其余               | 超时竞争、多源取最快 |
  | `withLatestFrom(b$)`      | 主流发值时，携带另一条流的最新值         | 主操作关联上下文数据 |
  | `startWith(value)`        | 给流一个初始值（在第一个真实值之前发出） | 设置默认状态         |

## combineLatest vs forkJoin

+ `combineLatest` vs `forkJoin` 经常搞混：

  + `combineLatest` — 任一流有新值就发，**适合持续变化的流**（如表单字段）
  + `forkJoin` — 等所有流 complete 后发一次，**适合一次性操作**（如并行 HTTP 请求）

+ combineLatest 适合持续的
+ forkJoin 适合一次性的

## 基本使用

+ merge

  ```js
  // merge ：合并，谁来了就立刻转发谁
  // 场景：多个按钮都能触发同一个动作
  rxjs.merge(
    rxjs.fromEvent(btn1, "click").pipe(rxjs.operators.map(() => "新建")),
    rxjs.fromEvent(btn2, "click").pipe(rxjs.operators.map(() => "删除"))
  ).subscribe(action => console.log("用户操作:", action));
  ```

+ combineLatest

  ```js
  // combineLatest：任一变化就重新组合
  // 场景：价格 × 数量 = 总价
  rxjs.combineLatest([price$, quantity$]).pipe(
    rxjs.operators.map(([price, qty]) => price * qty)
  ).subscribe(total => console.log("总价:", total));
  ```

+ forkJoin

  ```js
  // forkJoin ：全部完成才拿结果
  // 场景：页面初始化时同时请求用户信息和配置
  rxjs.forkJoin([
    rxjs.from(fetch("/api/user").then(r => r.json())),
    rxjs.from(fetch("/api/config").then(r => r.json()))
  ]).subscribe(([user, config]) => console.log("全部加载完成", user, config));
  ```

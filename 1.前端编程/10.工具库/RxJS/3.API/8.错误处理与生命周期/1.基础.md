# 基础

## 操作符

+ 错误处理与生命周期 —— "出问题了怎么办？"

  | 操作符                  | 用途                                     | 典型场景             |
  | ----------------------- | ---------------------------------------- | -------------------- |
  | `catchError(fn)`        | 拦截错误，返回替代流                     | 请求失败给默认值     |
  | `retry(n)`              | 失败后重试 N 次                          | 网络抖动自动重试     |
  | `retry({count, delay})` | 自定义重试策略（含延迟/指数退避）        | 指数退避重试         |
  | `timeout(ms)`           | 超时报错                                 | 请求超时控制         |
  | `finalize(fn)`          | 终止时执行（complete/error/unsubscribe） | 清理资源、关 loading |

> **注意：** 你可能在一些老教程或文章中看到 `retryWhen` 操作符，它在 RxJS 7 中已被标记为 **deprecated（废弃）**。官方推荐用 `retry()` 的配置对象形式替代，例如：`retry({ count: 3, delay: (err, retryCount) => timer(retryCount * 1000) })` 实现指数退避重试。

**`retry` 配置对象形式的完整示例：**

```js
switchMap(keyword =>
  from(fakeSearch(keyword)).pipe(
    // 重试 3 次，每次间隔递增：1s, 2s, 3s
    retry({
      count: 3,
      delay: function (err, retryCount) {
        console.log("第" + retryCount + "次重试...");
        return rxjs.timer(retryCount * 1000);
      }
    }),
    catchError(function (err) {
      console.warn("重试全部失败:", err.message);
      return of({ items: [] });
    })
  )
)
```

> 错误处理操作符的设计反映了一个架构原则——**局部错误不应该杀死全局流程**。在命令式代码中，一个未捕获的异常会终止整个执行；在 RxJS 管道中，你可以精确控制"在哪一层捕获、怎么降级、要不要重试"。这和微服务架构中的"熔断器模式"是同一种思路

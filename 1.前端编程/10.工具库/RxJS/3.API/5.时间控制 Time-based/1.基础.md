# 时间控制 Time-based

## 操作符

+ 操作符

  | 操作符             | 行为                             | 典型场景         |
  | ------------------ | -------------------------------- | ---------------- |
  | `debounceTime(ms)` | 安静 N ms 后放行最后一个值       | 搜索输入防抖     |
  | `throttleTime(ms)` | 每 N ms 最多放行一个值           | 滚动监听、resize |
  | `delay(ms)`        | 每个值延迟 N ms 后发出           | 动画、延迟显示   |
  | `auditTime(ms)`    | 每 N ms 发出该周期内最后一个值   | 性能监控采样     |
  | `sampleTime(ms)`   | 每 N ms 采样源流的最新值         | 固定频率采样     |
  | `bufferTime(ms)`   | 积攒 N ms 的值一次性发出（数组） | 批量日志上报     |

+ 时间控制操作符管理值的发送频率和时机

## auditTime vs sampleTime vs throttleTime 的区别

+ 这三个都是"限频率"的操作符，区别在于**取哪个值**：

  + `throttleTime(ms)` — 时间窗口开始时的**第一个值**（默认 leading）
  + `auditTime(ms)` — 时间窗口结束时的**最后一个值**
  + `sampleTime(ms)` — 固定间隔采样，不管源流有没有新值都会检查

+ 大多数场景用 `debounceTime` 或 `throttleTime` 就够了，`auditTime` 和 `sampleTime` 是更精细的调频工具

> 为什么 RxJS 要提供这么多时间操作符？因为**"时间"是异步编程中最难管理的维度**。用户的操作频率、网络的响应延迟、服务器的承受能力——这三者的节奏通常不一致。时间操作符的存在，是为了让你在这三个节奏之间做缓冲和协调

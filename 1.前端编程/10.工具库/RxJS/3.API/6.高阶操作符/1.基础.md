# 基础

## 解决的问题

+ 背压：当生产速度超过消费速度的时候怎么处理？

+ 后端其实有标准的解决手段：队列缓冲、拒绝新信息、生产者限速

+ 前端的生产者是用户操作和网络事件，你不能让用户操作慢点，也不能让服务器推送慢点，所以前端的背压策略就只能在消费端做

## 新任务和旧任务 的关系

+ 高阶操作符处理的是 **"Observable 中的 Observable"** ——当外层流的每个值都会触发一个新的异步流时，新旧流之间该怎么协调？

+ 上游流不断触发新的异步任务时，新任务和旧任务之间应该是什么关系

+ 这种关系可以划分为2个维度：

  + 维度一：旧任务要不要继续

    + 要   --> 保留旧任务
    + 不要 --> 取消旧任务

  + 维度二：新任务要不要立刻开始

    + 要   --> 立刻启动新任务
    + 不要 --> 推迟或者丢弃新任务

## 操作符

+ 操作符

  | 操作符       | 策略             | 一句话总结                     |
  | ------------ | ---------------- | ------------------------------ |
  | `switchMap`  | 切换（取消旧的） | 只要最新的结果                 |
  | `mergeMap`   | 并发（全都要）   | 每个任务都执行，交错输出       |
  | `concatMap`  | 排队（一个个来） | 严格保序，前一个完了才做后一个 |
  | `exhaustMap` | 忽略（忙着呢）   | 正在执行时新请求被丢弃         |

  |            | 新任务立刻执行  | 新任务推迟/丢弃                  |
  | ---------- | --------------- | -------------------------------- |
  | 保留旧任务 | mergeMap(并发)  | concatMap(排队)/exhaustMap(忽略) |
  | 取消旧任务 | switchMap(切换) | 无                               |

+ 这几个操作符解决了 Promise 链和 async/await **完全无法声明式处理** 的问题——新旧异步任务的关系
+ `await` 只能表达"等这一个完成再做下一个"（相当于 concatMap），你没有内建的方式说"取消上一个"或"正在忙就忽略新的"

## 使用场景

+ 搜索用 switchMap（旧的不要）
+ 保存用 concatMap（排队）
+ 批量用 mergeMap（并发）
+ 防重复用 exhaustMap（忙着就忽略）

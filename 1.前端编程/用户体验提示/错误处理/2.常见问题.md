# 常见问题

## 常见问题1 只写了成功路径

+ 先看一段很常见的代码：

  ```js
  async function handleSubmit(values: FormValues) {
    setLoading(true);

    try {
      await submitForm(values);
      message.success('提交成功');
      refreshList();
    } catch (error) {
      message.error('提交失败');
    } finally {
      setLoading(false);
    }
  }
  ```

+ 这段代码看起来没什么问题

  + 它有 `loading`

  + 它有 `try...catch`

  + 它有成功提示

  + 它也有失败提示

+ 如果只是本地自测，这段代码大概率能顺利通过

+ 但如果放到真实业务里，它其实处理得非常粗糙
+ 因为所有失败都被压缩成了一句话：

  + 提交失败

+ 但用户并不知道为什么失败。开发也不知道为什么失败

+ 更重要的是，不同失败其实应该有完全不同的处理方式

+ 比如：

  + 如果是表单字段不合法，应该把错误定位到字段上

  + 如果是余额不足，应该告诉用户具体业务原因

  + 如果是未登录，应该引导用户重新登录

  + 如果是网络超时，应该允许用户重试

  + 如果是权限不足，应该告诉用户没有操作权限

  + 如果是服务端异常，应该保留用户输入并提示稍后再试

+ 这些处理方式都不一样

+ 如果前端只用一个 catch 兜底，就等于把所有异常路径都抹平了

+ 这也是很多前端功能上线后不好用的原因：代码只关心成功时怎么走，没有认真设计失败时怎么收场


## 常见问题2 错误处理不是 catch 一下就完了

+ 很多人把错误处理理解成：

  ```js
  try {
    await request();
  } catch (error) {
    message.error('操作失败');
  }
  ```

+ 这当然算错误处理，但只算最基础的兜底

+ 真正的错误处理，至少要回答三个问题：

  1. 发生了什么？

  2. 用户现在还能做什么？

  3. 开发之后怎么排查？

+ 如果一个错误提示只能告诉用户“失败了”，但没有告诉用户为什么失败，也没有提供下一步动作，那这个错误处理只能算“通知失败”，不能算“处理失败”

+ 比如用户保存一篇文章时，接口返回失败

+ 一个粗糙的处理方式是：

  ```js
  message.error('保存失败');
  ```

+ 但用户接下来怎么办？

  + 是继续点保存？

  + 是刷新页面？

  + 是内容已经丢了？

  + 是网络问题还是权限问题？

+ 如果这些都不清楚，用户就会陷入一种很差的状态：知道失败了，但不知道该怎么恢复

+ 所以前端做错误处理，不只是把错误展示出来。

+ 更重要的是：让用户知道当前处境，并且知道下一步还能做什么

# 错误分类

## 错误分类

+ 错误分类1 用户输入错误
+ 错误分类2 业务规则错误
+ 错误分类3 权限错误
+ 错误分类4 网络错误
+ 错误分类5 服务端错误

### 错误分类1 用户输入错误

+ 用户输入错误是最常见的一类。

+ 比如：

  + 必填项没填

  + 手机号格式不对

  + 密码长度不够

  + 开始时间晚于结束时间

  + 上传文件类型不符合要求

+ 这类错误最好在前端就提前处理

+ 因为它们不需要等接口返回，也不应该让用户提交之后才发现。

+ 比如表单里某个字段错误，最好的处理方式不是弹一个全局提示：

  ```js
  表单填写错误
  ```

+ 而是把错误定位到具体字段：

  ```js
  手机号格式不正确
  ```

+ 用户输入错误的处理目标很明确：让用户知道哪里错了，以及怎么改

### 错误分类2 业务规则错误

+ 业务规则错误不是输入格式错了，而是当前业务状态不允许继续操作。

+ 比如：

  + 库存不足

  + 余额不足

  + 优惠券已过期

  + 订单已取消，不能再次支付

  + 审核通过后不能再修改

  + 当前状态不允许删除


+ 这类错误通常来自后端。

+ 因为业务规则最终要以后端判断为准。

+ 但前端不能只是把它当成普通接口失败。

+ 比如用户点击“删除订单”，后端返回：

  ```js
  {
    "code": "ORDER_HAS_BEEN_PAID",
    "message": "订单已支付，不能删除"
  }
  ```

+ 这时候前端如果只提示：

  ```js
  删除失败
  ```

+ 用户会很困惑。

+ 但如果提示：

  ```js
  订单已支付，不能删除
  ```

+ 用户至少知道这是业务规则限制，不是系统坏了

+ 业务规则错误的处理目标是：解释为什么不能做，而不是只告诉用户没做成

### 错误分类3 权限错误

+ 权限错误也很常见

+ 比如：

  + 用户未登录

  + token 过期

  + 当前账号没有操作权限

  + 当前角色不能访问某个页面

  + 企业成员被移除

+ 权限错误不能只靠前端隐藏按钮解决

+ 按钮隐藏只是体验优化，真正的权限校验一定要在后端

+ 前端要做的是根据不同权限错误给出合适的引导

+ 比如：

  + 未登录：跳转登录页

  + token 过期：尝试刷新 token 或重新登录

  + 无权限：展示无权限页面

  + 权限被回收：提示用户联系管理员

+ 这些场景如果都统一提示“请求失败”，体验会非常差

+ 权限错误的处理目标是：明确用户身份或权限状态，并引导到正确路径

### 错误分类4 网络错误

+ 网络错误和业务错误不一样。

+ 业务错误说明系统知道发生了什么，只是不允许继续。

+ 网络错误则是请求根本没有稳定完成。

+ 比如：

  + 断网

  + 超时

  + DNS 失败

  + 请求被取消

  + 浏览器从后台恢复后连接失效

+ 这类错误最需要考虑重试和恢复。

  + 比如一个列表加载失败，可以展示一个“重新加载”按钮

  + 一个表单提交超时，应该保留表单内容，让用户可以再次提交。

  + 一个文件上传失败，应该保留文件选择状态，允许重新上传

+ 网络错误的处理目标是：不要让用户因为一次连接失败，就丢失当前操作

### 错误分类5 服务端错误

+ 服务端错误通常指 500、接口异常、数据库异常、服务不可用等问题。

+ 这类错误对用户来说很难理解。

+ 用户不关心数据库连接失败，也不关心某个服务挂了。

+ 用户关心的是：

  + 当前操作有没有成功？

  + 内容有没有丢？

  + 还能不能重试？

  + 是否需要稍后再试？

+ 所以服务端错误不应该暴露太多内部细节。

+ 比如不要直接把堆栈信息展示给用户。

+ 但对开发来说，服务端错误又必须足够可排查。

+ 这就要求前端最好能记录必要信息：

  + 请求地址

  + 请求参数

  + 状态码

  + 错误码

  + requestId 或 traceId

  + 用户当前操作路径

+ 服务端错误的处理目标是双重的：对用户隐藏复杂细节，对开发保留排查线索

### 错误分类5 前端运行时错误

+ 还有一类错误发生在前端自己身上。

+ 比如：

  + 读取空对象属性

  + 组件渲染崩溃

  + 第三方 SDK 报错

  + 数据结构变化导致页面异常

  + 异步回调里访问了已经卸载的组件状态

+ 这类错误如果没有兜底，很容易导致整个页面白屏

+ React 里可以用 Error Boundary 做组件级兜底

+ Vue 里也有全局错误处理和组件错误捕获机制

+ 但要注意，错误边界不是为了把错误吞掉

+ 它的作用是：让局部错误不要拖垮整个页面，同时把错误记录下来

+ 前端运行时错误的处理目标是：保护页面可用性，并让问题能被发现

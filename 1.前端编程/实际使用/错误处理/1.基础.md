# 错误处理

## 问题

+ 很多前端同学写需求时，最先关注的是正常流程

+ 比如一个表单提交功能，最直觉的实现路径是：

  + 用户填写表单

  + 点击提交按钮

  + 前端调用接口

  + 接口返回成功

  + 页面提示提交成功

  + 刷新列表或者跳转页面

+ 这条路径当然要写，而且必须写对

+ 但真实项目里，最容易出问题的往往不是这条路径

+ 真正麻烦的是：

  + 接口失败了怎么办？

  + 网络很慢怎么办？

  + 用户重复点击怎么办？

  + 请求还没回来，用户切页面了怎么办？

  + 服务端返回的数据结构不符合预期怎么办？

  + token 过期了怎么办？

  + 权限不足怎么办？

  + 用户提交一半断网了怎么办？

+ 这些问题都属于错误处理

+ 很多前端代码看起来能跑，但上线之后体验很脆弱，原因不是主流程没写通，而是异常路径没有设计

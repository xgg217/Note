# 错误处理应该成为状态设计的一部分

## 概述

+ 很多前端写请求状态时，只设计三个状态：

  ```
  idle -> loading -> success / error
  ```

+ 这在简单场景下够用。

+ 但真实业务往往没这么简单。

+ 比如一个提交表单，状态可能是：

  ```
                            UI展示或响应
  idle
    -> validating
    -> submitting
    -> success
    -> validationError      定位到具体字段
    -> businessError        展示业务原因
    -> networkError         提供重试
    -> unauthorized         引导登录
    -> serverError          保留内容并提示稍后再试
    -> canceled             用户主动取消，不一定要提示错误
    -> stale                旧请求返回了过期结果，不应该覆盖新状态
  ```

+ 这些状态不是为了让代码看起来复杂，而是为了让 UI 能做出正确响应


+ 如果所有错误都只用一个 `error: true` 表示，页面就只能做一种处理

+ 这也是很多前端状态设计的问题：状态太粗，导致错误处理只能粗糙

+ 错误处理应该是状态设计的一部分

+ 也就是说，不是请求失败之后临时补一个 message，而是在设计功能时就想清楚：

  + 成功时页面进入什么状态？

  + 失败时页面进入什么状态？

  + 用户能不能重试？

  + 用户输入是否保留？

  + 是否需要回滚界面？

  + 是否需要重新拉取数据？

  + 取消请求算不算错误？

  + 旧请求返回时是否还允许更新页面？

+ 如果这些问题提前想清楚，代码会更稳定，用户体验也会更完整

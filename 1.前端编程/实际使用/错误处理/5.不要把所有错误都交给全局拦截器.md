# 不要把所有错误都交给全局拦截器

## 概述

+ 很多项目都会封装 request 方法

+ 然后在里面统一处理错误：

  ```js
  request.interceptors.response.use(
    (response) => response.data,
    (error) => {
      if (error.status === 401) {
        redirectToLogin();
        return;
      }

      message.error(error.message || '请求失败');
      return Promise.reject(error);
    }
  );
  ```

+ 这种全局处理是有必要的

+ 因为有些错误确实应该统一处理

+ 比如：

  + token 失效

  + 服务不可用

  + 通用网络异常

  + 未知错误兜底


+ 但全局错误处理有一个边界：它只能处理通用错误，不能替代业务错误处理。

+ 因为很多错误只有具体业务场景才知道怎么处理。

+ 比如导入 Excel 失败。全局提示一句“导入失败”没有什么价值。更好的处理可能是：

  + 展示失败行号

  + 展示失败原因

  + 允许下载错误报告

  + 保留用户上传的文件

  + 允许修正后重新导入

+ 再比如 AI 生成失败。全局提示一句“生成失败”也不够。更好的处理可能是：

  + 保留用户原输入

  + 保留已经生成的部分内容

  + 允许重新生成

  + 允许切换提示词

  + 告诉用户是不是上下文过长

+ 所以全局拦截器应该做的是兜底和分发，而不是把所有错误都变成统一弹窗

+ 一个更合理的原则是：越靠近业务的错误，越应该由业务自己处理
